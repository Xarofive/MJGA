## Зачем нужен повтор

В Base вы учились писать первые программы: выводить текст, хранить значения, вызывать методы, проверять условия, повторять действия в циклах, работать с массивами, строками и простыми коллекциями. Там было важно привыкнуть к синтаксису и увидеть, что Java-код можно запускать и отлаживать.

В текущем курсе фокус меняется. Теперь недостаточно просто получить правильный ответ на одном примере. Нужно писать код так, чтобы его можно было читать, расширять, проверять и безопасно менять. Поэтому первый модуль начинается с повторения, но это не пересказ Base. Мы проверяем, умеете ли вы соединять темы Base в одну маленькую программу и постепенно улучшать ее структуру.

Главная тема этой лекции:

```text
Base: я могу написать решение
↓
Следующий уровень: я могу улучшить, организовать и проверить решение
```

Слово "улучшить" здесь не означает "переписать все красиво ради красоты". В программировании для этого есть важный термин - **рефакторинг**.

**Рефакторинг** - это изменение структуры кода без изменения поведения программы. То есть программа делает то же самое, но код становится понятнее, безопаснее или удобнее для дальнейших изменений.

Простой пример:

```java
int result = price - price * discount / 100;
```

После рефакторинга:

```java
// Название метода показывает, какое действие выполняется.
// Расчет вынесен из общей строки в отдельное место.
int result = calculateFinalPrice(price, discount);
```

Результат тот же, но смысл стал виден по имени метода.

Пример из Base:

```java
int[] scores = {90, 70, 100};
int sum = 0;

for (int i = 0; i < scores.length; i++) {
    int score = scores[i];
    sum += score;
}

double average = (double) sum / scores.length;
System.out.println(average);
```

В Base здесь было важно понять массив, цикл и деление. В текущем курсе к этому добавляются вопросы разработчика:

- Что будет, если массив пустой?
- Почему нужен cast к `double`?
- Как вынести расчет в метод?
- Как назвать метод так, чтобы смысл был понятен?
- Как проверить код через Debugger?

Это не новая тема вместо Base, а следующий слой поверх уже знакомого материала: мы берем тот же код и учимся делать его понятнее и надежнее.

## Первый разговор о кодстайле

Когда код становится больше одного учебного примера, важен не только результат, но и стиль написания. **Кодстайл** - это набор правил, которые помогают писать код единообразно: как называть переменные, где ставить пробелы, как оформлять блоки, как располагать классы и методы.

Кодстайл не делает программу "умнее", но делает ее понятнее для человека. Это важно, потому что разработчик чаще читает и меняет код, чем пишет его с нуля.

**Важно:** начиная с этого курса, задачи будут проверяться платформой не только на правильный результат, но и на соблюдение кодстайла. Если решение работает, но оформлено не по правилам, платформа может не принять задачу до исправления стиля.

В текущем курсе мы будем возвращаться к кодстайлу постепенно. Сейчас достаточно запомнить базовый минимум:

- имена переменных и методов пишем в `lowerCamelCase`: `finalPrice`, `calculateTotal`;
- имена классов пишем в `UpperCamelCase`: `Main`, `OrderService`;
- имя должно объяснять смысл, а не просто быть коротким: `price` лучше, чем `p`;
- фигурные скобки у `if`, `for`, методов и классов пишем явно;
- один метод должен делать одно понятное действие;
- комментарии нужны не для пересказа каждой строки, а для пояснения смысла, причины или учебного акцента.

Пример:

```java
// Неудачное имя: непонятно, что хранит переменная.
int x = 1200;

// Лучше: имя объясняет смысл значения.
int orderPrice = 1200;
```

## Что берем с собой из Base

### Метод `main`

`main` остается точкой входа для консольных программ:

```java
public class Main {
    public static void main(String[] args) {
        System.out.println("Current course starts here");
    }
}
```

В текущем курсе `main` постепенно станет короче: он будет запускать программу, а основная логика будет переезжать в методы и классы. Так код проще читать, проверять, исправлять и развивать.

Как часто писали в Base:

```java
public static void main(String[] args) {
    int price = 1000;
    int discount = 10;
    // Расчет находится прямо внутри main.
    // Пока пример маленький, это терпимо, но при росте программы main быстро перегружается.
    int finalPrice = price - price * discount / 100;
    System.out.println(finalPrice);
}
```

Код работает, но расчет "спрятан" внутри `main`. Если такой расчет понадобится в другом месте, его придется копировать. Если в расчете появится ошибка, ее будет сложнее проверять отдельно.

После улучшения:

```java
public static void main(String[] args) {
    // main теперь только запускает сценарий: вызывает расчет и выводит результат.
    int finalPrice = calculateFinalPrice(1000, 10);
    System.out.println(finalPrice);
}

// Название метода отражает действие: посчитать финальную цену.
// Логика расчета выделена отдельно, поэтому ее проще читать, проверять и переиспользовать.
public static int calculateFinalPrice(int price, int discountPercent) {
    return price - price * discountPercent / 100;
}
```

Это небольшой рефакторинг: поведение программы осталось тем же, но расчет получил имя и отдельное место. Теперь метод `calculateFinalPrice` можно вызвать повторно, проверить отдельно и отладить отдельно.

### Методы

Метод должен иметь понятную **ответственность**.

Ответственность метода - это ответ на вопрос: "За что этот метод отвечает?" Если метод одновременно читает ввод, считает скидку, печатает результат и меняет коллекцию, он отвечает сразу за слишком многое. Такой метод трудно проверять и трудно менять.

Пример метода с понятной ответственностью:

```java
public static int countPositiveNumbers(int[] numbers) {
    int count = 0;

    for (int i = 0; i < numbers.length; i++) {
        int number = numbers[i];

        if (number > 0) {
            count++;
        }
    }

    return count;
}
```

У метода понятный вход, понятный выход и одно действие.

### Условия

В Base вы изучали `if`, `else if`, `else`. В текущем курсе условия будут не только выбирать ветку кода, но и защищать правила программы.

Здесь появляется термин **предметная область**.

Предметная область - это часть реального мира, которую описывает программа. Если мы пишем программу для доставки, предметная область - это заказы, товары, вес, расстояние, цена доставки, скидки и ограничения. Если мы пишем программу для учебного журнала, предметная область - это студенты, оценки, группы и средний балл.

**Правила предметной области** - это ограничения, которые должны выполняться по смыслу задачи:

- количество товара не может быть меньше нуля;
- скидка не может быть больше 100%;
- заказ без товаров нельзя оформить;
- возраст студента не может быть отрицательным;
- расстояние доставки не может быть меньше нуля.

Такие правила часто проверяются через `if`.

```java
public static boolean canCreateOrder(int quantity, int balance) {
    if (quantity <= 0) {
        return false;
    }

    if (balance <= 0) {
        return false;
    }

    return true;
}
```

Такой стиль называется ранним выходом. Он помогает не строить глубокие вложенные `if`.

### Циклы

Циклы в текущем курсе остаются основой. Позже появятся более короткие способы обработки коллекций, но обычный цикл все равно нужно понимать: именно он показывает, что происходит с каждым элементом.

```java
public static int findMax(int[] numbers) {
    if (numbers.length == 0) {
        return 0;
    }

    int max = numbers[0];

    for (int i = 1; i < numbers.length; i++) {
        int number = numbers[i];

        if (number > max) {
            max = number;
        }
    }

    return max;
}
```

Здесь есть важная деталь текущего курса: пустой массив обработан явно. Такой случай называют **крайним случаем**.

Крайний случай - это входные данные, которые находятся на границе обычного сценария:

- пустой массив;
- строка без символов;
- список из одного элемента;
- нулевая цена;
- отрицательное число там, где ожидается положительное;
- `null`, если метод принимает объект.

Код, который работает только на удобных данных, часто ломается на крайних случаях.

### Массивы и строки

Массив хранит фиксированное количество элементов. Строка хранит текст и является объектом.

```java
String name = "Alex";
int length = name.length();
char first = name.charAt(0);
```

Из Base важно помнить: `String` сравнивают через `equals`, а не через `==`.

```java
String role = "admin";

if (role.equals("admin")) {
    System.out.println("Access granted");
}
```

Позже в текущем курсе эта идея станет основой для `equals` у собственных классов. Сейчас важно помнить Base-правило: строки сравниваем по содержимому, а не по ссылке.

### `ArrayList` и `HashMap`

В Base вы использовали `ArrayList` как изменяемый список, а `HashMap` как хранилище по ключу.

```java
ArrayList<String> names = new ArrayList<>();
names.add("Anna");
names.add("Ivan");

HashMap<String, Integer> scores = new HashMap<>();
scores.put("Anna", 90);
scores.put("Ivan", 75);
```

В текущем курсе мы вернемся к коллекциям глубже. Пока не нужно знать все будущие термины. Сейчас важно проверить, что базовая работа со списком и словарем не вызывает проблем:

- добавить элемент в список;
- пройти по списку циклом;
- положить значение в `HashMap`;
- получить значение по ключу;
- проверить, что ключ существует.

## Как выглядит следующий уровень мышления на простой задаче

Задача: есть массив оценок, нужно посчитать среднюю оценку только для положительных значений.

Как могли написать после Base:

```java
public static double averagePositive(int[] grades) {
    int sum = 0;
    int count = 0;

    for (int i = 0; i < grades.length; i++) {
        int grade = grades[i];

        if (grade > 0) {
            sum += grade;
            count++;
        }
    }

    // Здесь есть скрытая проблема: если положительных оценок нет,
    // count останется равным 0.
    return (double) sum / count;
}
```

На хорошем входе код работает. Но теперь мы задаем следующий вопрос: что будет, если положительных оценок нет? Тогда `count` равен `0`, и результат будет некорректным.

После улучшения:

```java
public static double averagePositive(int[] grades) {
    int sum = 0;
    int count = 0;

    for (int i = 0; i < grades.length; i++) {
        int grade = grades[i];

        if (grade > 0) {
            sum += grade;
            count++;
        }
    }

    // Ситуация без положительных оценок обработана явно: деления на 0 не будет.
    if (count == 0) {
        return 0.0;
    }

    return (double) sum / count;
}
```

Здесь мы явно описали поведение для крайнего случая. В текущем курсе такая точность будет появляться все чаще: программа должна быть понятной не только на идеальных данных.

## Debugger как рабочий инструмент

В Base Debugger был отдельной темой. В текущем курсе он становится обычной частью работы.

Мини-порядок:

1. Поставьте breakpoint на строку, где начинается подозрительный расчет.
2. Запустите программу в режиме Debug.
3. Идите по шагам через Step Over.
4. Следите за значениями переменных.
5. Сравнивайте ожидаемое значение с фактическим.

Пример кода для отладки:

```java
public static int calculateTotal(int[] prices) {
    int total = 0;

    for (int i = 0; i < prices.length; i++) {
        int price = prices[i];

        if (price > 0) {
            // Поставьте breakpoint на следующую строку.
            // Так вы увидите, что total не накапливает сумму, а заменяется текущей ценой.
            total = price;
        }
    }

    return total;
}
```

Ошибка здесь в строке `total = price;`. Мы не накапливаем сумму, а каждый раз заменяем ее последней ценой. Debugger помогает увидеть, как `total` меняется на каждой итерации.

Правильно:

```java
total += price;
```

## Частые пробелы после Base

### Код работает только на одном примере

Если программа проверена только на одном наборе данных, это не значит, что она правильная.

Проверяйте минимум:

- обычный случай;
- пустой массив или пустую строку;
- одно значение;
- отрицательные числа, если они возможны;
- `null`, если метод принимает ссылочный тип.

### Слишком много логики в `main`

`main` должен запускать сценарий, а не быть местом для всего кода. В текущем курсе логика будет уходить в методы и классы. Позже появятся отдельные классы для расчетов, хранения данных и управления действиями приложения.

### Неясные имена

Имена `a`, `b`, `x`, `map`, `list` допустимы в очень коротких примерах, но плохо работают в учебном проекте.

Лучше:

```java
int orderCount = 5;
double deliveryPrice = 250.0;
HashMap<String, Integer> productQuantities = new HashMap<>();
```

### Игнорирование ошибок

Если метод может получить некорректные данные, нужно решить, что делать:

- вернуть специальное значение;
- вывести сообщение;
- выбросить исключение;
- запретить создание некорректного объекта.

В Base мы часто выбирали простой вариант. В текущем курсе будем учиться выбирать осознанно.

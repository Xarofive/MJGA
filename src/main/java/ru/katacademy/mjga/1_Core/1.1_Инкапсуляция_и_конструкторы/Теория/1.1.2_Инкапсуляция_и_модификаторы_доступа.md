# Инкапсуляция и модификаторы доступа

## TL;DR

- Инкапсуляция скрывает внутреннее состояние и оставляет внешнему коду разрешенные операции.
- `public` открывает элемент для использования из других классов.
- `private` разрешает доступ только внутри того же класса.
- Package-private ограничивает доступ текущим пакетом и записывается без ключевого слова.
- `protected` открывает элемент текущему пакету и наследникам.
- Публичный контракт - доступные внешнему коду операции класса.
- Поля обычно делают `private`, чтобы нельзя было обойти проверки.
- Закрытые поля не помогают, если публичные методы изменения принимают любые значения.

## Что продолжаем из предыдущей лекции

Мы уже выяснили, что класс должен хранить правила рядом со своим состоянием. Но пока поля доступны напрямую, внешний код может не вызывать безопасные методы и записать недопустимое значение.

Реализация - конкретный код и структура данных, с помощью которых класс
выполняет работу.

Теперь нужно провести техническую границу: определить, какие части класса
доступны снаружи, а какие остаются его внутренней реализацией.

## Зачем это нужно

Без ограничения доступа класс не управляет собственным состоянием. Любой участок программы может изменить поле в обход проверки. Кроме того, внешний код начинает зависеть от устройства класса и ломается при изменении полей.

Инкапсуляция уменьшает количество деталей, которые нужно помнить при использовании объекта. Внешний код знает, что можно сделать, но не обязан знать, как именно это реализовано.

## Проблема открытого состояния

В Base поля классов объявлялись без модификатора доступа:

```java
class BankAccount {
    // Нет модификатора: поля доступны всему пакету.
    String owner;
    long balanceInCents;
}
```

Такой класс удобно использовать в первых задачах:

```java
BankAccount account = new BankAccount();

// Прямая запись не проходит никакую проверку.
account.owner = "Alice";
account.balanceInCents = 100_000L;
```

Но любой код может записать любое значение:

```java
// Открытые поля позволяют создать некорректное состояние.
account.owner = null;
account.balanceInCents = -5_000_000L;
```

Если у класса есть правила, прямой доступ к полям позволяет эти правила обойти.

## Что такое инкапсуляция

Инкапсуляция - это объединение данных и операций над ними внутри класса с ограничением прямого доступа к внутреннему состоянию.

У инкапсуляции две части:

1. Данные и правила находятся рядом, внутри одного класса.
2. Внешний код работает с объектом через разрешенные операции, а не изменяет его устройство напрямую.

Инкапсуляция не означает «сделать все `private` и написать методы для каждого поля». Ее цель - защитить правила объекта и уменьшить количество деталей, которые должен знать внешний код.

### Аналогия из жизни

Инкапсуляция похожа на работу банкомата. Клиент может выбрать разрешенную
операцию: посмотреть баланс, внести или снять деньги. При этом он не получает
доступ к кассетам с купюрами и внутреннему механизму банкомата. Объект также
открывает операции, но скрывает прямой доступ к своему устройству.

## Внутреннее состояние и публичный контракт

Внутреннее состояние - это данные и детали реализации, которыми класс управляет самостоятельно.

Публичный контракт класса - это доступные внешнему коду способы работы с объектом.

Слово «контракт» здесь означает обещание класса:

- какие операции доступны;
- какие аргументы они принимают;
- какой результат возвращают;
- какие правила соблюдают.

Например:

```java
class BankAccount {
    // Изменение: поля стали private.
    // Теперь ими управляет только BankAccount.
    private String owner;
    private long balanceInCents;

    // Публичная операция входит во внешний контракт.
    public boolean deposit(long amountInCents) {
        if (amountInCents <= 0) {
            return false;
        }

        balanceInCents += amountInCents;
        return true;
    }

    // Публичная операция сохраняет правило баланса.
    public boolean withdraw(long amountInCents) {
        if (amountInCents <= 0 || amountInCents > balanceInCents) {
            return false;
        }

        balanceInCents -= amountInCents;
        return true;
    }

    // Читать баланс разрешено, записывать напрямую - нет.
    public long currentBalanceInCents() {
        return balanceInCents;
    }
}
```

Внутреннее состояние:

```java
private String owner;
private long balanceInCents;
```

Публичный контракт:

```java
deposit(...)
withdraw(...)
currentBalanceInCents()
```

Внешний код может попросить счет выполнить операцию, но не может напрямую установить баланс.

### Аналогия из жизни

Публичный контракт похож на меню ресторана. Посетитель видит, какие блюда
можно заказать и что он получит, но не управляет холодильниками, плитой и
работой поваров. Методы публичного контракта так же описывают доступные
операции, не раскрывая внутреннюю реализацию класса.

## Модификаторы доступа

Модификатор доступа - ключевое слово, которое определяет, откуда можно
обратиться к классу, полю или методу.

В Java есть четыре уровня доступа:

- `public`;
- `protected`;
- `private`;
- package-private.

В этом модуле основное внимание уделяется `public`, `private` и
package-private. `protected` тесно связан с наследованием, поэтому здесь
дается его общее правило, а подробности будут разобраны в модуле о
наследовании.

### Аналогия из жизни

Модификаторы доступа можно сравнить с зонами в офисе:

- `public` - зона для всех посетителей;
- package-private - помещение для сотрудников одного отдела;
- `protected` - зона для отдела и сотрудников связанных подразделений;
- `private` - личный кабинет, доступный только владельцу.

Аналогия помогает увидеть ширину доступа, но точные правила всегда задаются
самим языком Java.

## `public`

`public` означает, что элемент доступен из любого класса, если сам класс виден и корректно импортирован.

```java
public class BankAccount {
    public boolean deposit(long amountInCents) {
        // ...
        return true;
    }
}
```

Публичными обычно делают операции, которые входят в контракт класса.

Важно: `public` не означает «хороший» или «главный». Это означает «разрешенный для внешнего использования».

Чем больше публичных элементов, тем больше способов использовать класс. Каждый такой способ придется поддерживать и учитывать при изменениях.

## `private`

`private` означает, что элемент доступен только внутри того же класса.

```java
public class BankAccount {
    private long balanceInCents;

    public long currentBalanceInCents() {
        return balanceInCents;
    }
}
```

Этот код снаружи класса не скомпилируется:

```java
BankAccount account = new BankAccount();

// Ошибка компиляции: private-поле недоступно снаружи.
account.balanceInCents = -100; // ошибка компиляции
```

Ошибка возникает до запуска программы. Компилятор не разрешает нарушить границу доступа.

`private` можно применять не только к полям:

```java
public class PriceCalculator {
    public long calculate(long priceInCents, int quantity) {
        // Публичный метод использует внутреннюю проверку.
        if (!isValidQuantity(quantity)) {
            return 0;
        }
        return priceInCents * quantity;
    }

    // Метод нужен только самому классу, поэтому он private.
    private boolean isValidQuantity(int quantity) {
        return quantity > 0;
    }
}
```

Метод `isValidQuantity` нужен только как внутренняя деталь `PriceCalculator`, поэтому внешний код не должен его вызывать.

## Package-private

Package-private - доступ только из классов того же пакета.

Для package-private нет отдельного ключевого слова. Нужно просто не писать модификатор:

```java
// Нет модификатора у класса и метода:
// они доступны только внутри пакета.
class PricePolicy {
    long calculateDiscount(long totalInCents) {
        return totalInCents / 20;
    }
}
```

И класс `PricePolicy`, и его метод доступны другим классам того же пакета, но недоступны из другого пакета.

Название package-private читается как «закрытый пределами пакета».

Пример:

```text
shop.pricing.PricePolicy
shop.pricing.DiscountCalculator
shop.app.Main
```

`DiscountCalculator` из пакета `shop.pricing` сможет обратиться к package-private классу `PricePolicy`. Класс `Main` из `shop.app` - не сможет.

Package-private полезен для деталей, которыми пользуются несколько соседних классов, но которые не должны становиться частью внешнего контракта пакета.

## `protected`

`protected` разрешает доступ:

- внутри самого класса;
- из любого класса того же пакета;
- из класса-наследника, даже если наследник находится в другом пакете.

```java
public class Account {
    // Поле доступно классам того же пакета и наследникам.
    protected long balanceInCents;
}
```

Наследник - класс, который получает поля и методы другого класса через
механизм наследования.

В другом пакете `protected` не делает элемент доступным для любого объекта.
Доступ получает именно код наследника и только по правилам наследования.
Подробно это ограничение будет рассмотрено после знакомства с `extends`.

Для обычного внутреннего состояния класса предпочтительной отправной точкой
остается `private`. Не следует заменять его на `protected` без необходимости.

## Таблица доступа

| Модификатор | Тот же класс | Тот же пакет | Наследник в другом пакете | Другой класс в другом пакете |
| --- | --- | --- | --- | --- |
| `private` | Да | Нет | Нет | Нет |
| package-private | Да | Да | Нет | Нет |
| `protected` | Да | Да | Да* | Нет |
| `public` | Да | Да | Да | Да |

`*` В другом пакете доступ к `protected`-элементу связан с наследованием и
имеет дополнительные ограничения. Они будут разобраны в модуле о
наследовании.

## Доступ к классу и доступ к его членам

Поля и методы класса называют членами класса.

Доступ проверяется поэтапно:

1. Видит ли код сам класс?
2. Видит ли код нужный член класса?

Например:

```java
class InternalCalculator {
    public long calculate(long value) {
        return value * 2;
    }
}
```

Метод объявлен `public`, но класс package-private. Из другого пакета нельзя использовать `InternalCalculator`, поэтому публичность метода не помогает.

Для верхнеуровневого класса, объявленного прямо в отдельном файле, обычно используются:

- `public`;
- package-private.

`private` верхнеуровневый класс объявить нельзя.

## Сокрытие информации

Сокрытие информации - принцип, по которому класс показывает только необходимый способ использования и скрывает детали реализации.

Пусть корзина сначала хранит итоговую сумму в поле:

```java
class Cart {
    // Реализация скрыта: внешний код не работает с полем напрямую.
    private long totalInCents;

    // Контракт сохранится, даже если расчет суммы изменится.
    public long currentTotalInCents() {
        return totalInCents;
    }
}
```

Позже реализация может измениться: корзина будет хранить товары и вычислять
сумму при запросе. Если внешний код использует только
`currentTotalInCents()`, его не придется переписывать.

Если же внешний код напрямую работает с полем, внутреннее изменение становится сложнее.

## Минимальный публичный контракт

`public` должны быть те методы, которые внешний код действительно должен вызывать.
Само наличие публичных методов не является проблемой: без них объектом нельзя
будет пользоваться из других классов. Проблема возникает, когда наружу
открывают внутренние проверки и слишком универсальные операции «на всякий
случай».

Плохо:

```java
public class BankAccount {
    // Поле уже private, но этого недостаточно.
    private long balanceInCents;

    // Плохо: внутренняя проверка открыта наружу.
    public boolean isAmountPositive(long amount) {
        return amount > 0;
    }

    // Плохо: универсальный метод позволяет обойти смысл операций.
    public void changeBalance(long delta) {
        balanceInCents += delta;
    }
}
```

Внешний код получает слишком много возможностей:

- менять баланс любым положительным или отрицательным `delta`;
- вызывать техническую проверку `isAmountPositive`, которая нужна самому классу;
- обходить отдельные правила пополнения и снятия денег.

Лучше:

```java
public class BankAccount {
    // Состояние по-прежнему закрыто.
    private long balanceInCents;

    // Оставляем public: это разрешенная операция счета.
    public boolean deposit(long amountInCents) {
        if (!isAmountPositive(amountInCents)) {
            return false;
        }

        balanceInCents += amountInCents;
        return true;
    }

    // Оставляем public: внешний код должен уметь снимать деньги.
    public boolean withdraw(long amountInCents) {
        if (amountInCents <= 0 || amountInCents > balanceInCents) {
            return false;
        }

        balanceInCents -= amountInCents;
        return true;
    }

    // Оставляем public только чтение баланса.
    public long currentBalanceInCents() {
        return balanceInCents;
    }

    // Изменение: техническая проверка стала private.
    private boolean isAmountPositive(long amountInCents) {
        return amountInCents > 0;
    }
}
```

Методы `deposit`, `withdraw` и `currentBalanceInCents` остаются `public`, потому что
они входят в публичный контракт счета. Метод `isAmountPositive` становится
`private`, потому что это внутренняя проверка. Универсальный метод
`changeBalance` удален: он позволял менять баланс без учета смысла операции.

## Закрытых полей недостаточно

Такой класс формально имеет закрытые поля:

```java
class Product {
    private long priceInCents;

    // Читать цену разрешено.
    public long currentPriceInCents() {
        return priceInCents;
    }

    // Проблема: публичный метод изменения принимает любое значение.
    public void changePrice(long newPriceInCents) {
        priceInCents = newPriceInCents;
    }
}
```

Но метод `changePrice` снова разрешает записать отрицательную цену:

```java
// Инкапсуляция формальная:
// публичный метод пропускает отрицательную цену.
product.changePrice(-100);
```

Поле закрыто синтаксически, но правило не защищено.

Настоящая инкапсуляция требует осмысленного публичного контракта:

```java
// Лучше: название описывает действие,
// а метод защищает правило цены.
public boolean changePrice(long newPriceInCents) {
    if (newPriceInCents < 0) {
        return false;
    }

    priceInCents = newPriceInCents;
    return true;
}
```

Название метода выражает действие, а проверка защищает состояние.

## Почему поля обычно `private`

Закрытые поля дают классу контроль:

- над допустимыми значениями;
- над способом изменения;
- над тем, какие данные можно читать;
- над будущим изменением внутренней реализации.

Это не абсолютный закон языка Java, но надежная отправная точка:

```java
private String name;
private long priceInCents;
private int quantity;
```

После этого для каждого поля нужно отдельно решить:

- нужно ли внешнему коду читать значение;
- можно ли менять значение после создания объекта;
- какая операция должна выполнять изменение;
- какие проверки обязательны.

## Частые ошибки

### Ошибка 1. Делать все элементы `public`

Чем шире публичный контракт, тем больше способов использовать класс неправильно и тем сложнее менять его реализацию.

### Ошибка 2. Считать `private` запретом на любое чтение

Поле может быть закрытым, а нужное значение доступно через контролируемый публичный метод.

### Ошибка 3. Путать package-private с `private`

Элемент без модификатора доступен всем классам того же пакета, а не только текущему классу.

### Ошибка 4. Закрыть поля и открыть небезопасные методы изменения

Если публичный метод разрешает записать любое значение, правила объекта
по-прежнему можно нарушить.

- Как записывается package-private?
- Почему `public` метод внутри package-private класса недоступен из другого пакета?
- Почему минимальный публичный контракт проще поддерживать?
- Достаточно ли сделать поля `private`, чтобы полностью защитить объект?

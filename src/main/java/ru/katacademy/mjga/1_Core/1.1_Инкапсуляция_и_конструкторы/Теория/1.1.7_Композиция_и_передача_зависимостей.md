# Композиция и передача зависимостей

## TL;DR

- Композиция строит сложный объект из других объектов.
- Поле объектного типа выражает связь «один объект имеет другой».
- Отдельный объект нужен, когда группа данных имеет собственный смысл и правила.
- Зависимость - объект, необходимый другому объекту для работы.
- Обязательную зависимость можно передать через конструктор.
- Обязательную зависимость нужно проверить и обычно сохранить в `final` поле.
- Точка сборки создает объекты и связывает их между собой.

## Что продолжаем из предыдущих лекций

Мы научились создавать отдельные корректные классы: закрывать поля, сохранять инварианты, требовать обязательные данные в конструкторах и фиксировать неизменяемые ссылки через `final`.

Теперь эти классы нужно связать. Покупатель использует адрес, заказ связан с покупателем, а один калькулятор может использовать другой калькулятор.

## Зачем это нужно

Если хранить все данные в одном классе, его ответственность постоянно растет. Если каждый класс самостоятельно создает всех помощников, связи становятся скрытыми и жесткими.

Композиция позволяет собирать программу из небольших объектов. Передача зависимостей через конструктор делает требования класса явными и гарантирует, что объект готов к работе сразу после создания.

## Когда одного класса недостаточно

Реальные модели состоят из нескольких связанных объектов.

У покупателя есть адрес:

```text
Customer -> Address
```

Заказ содержит покупателя и адрес доставки:

```text
Order -> Customer
Order -> Address
```

Калькулятор заказа использует правило скидки:

```text
OrderCalculator -> DiscountCalculator
```

Если поместить все данные и операции в один класс, он быстро станет слишком большим. Вместо этого программа строится из небольших объектов, каждый из которых отвечает за свою часть работы.

## Композиция

Композиция - построение более сложного объекта из других объектов.

### Аналогия из жизни

Композиция похожа на автомобиль, собранный из двигателя, колес, аккумулятора
и других узлов. Автомобиль использует эти части, но не является двигателем
или колесом. Так же сложный объект содержит другие объекты и поручает каждому
из них собственную часть работы.

В коде композиция обычно выражается полем объектного типа:

```java
public class Customer {
    private final String name;

    // Композиция: Customer хранит ссылку на Address.
    private final Address address;

    // Связанный объект передается при создании Customer.
    public Customer(String name, Address address) {
        this.name = name;
        this.address = address;
    }
}
```

`Customer` содержит ссылку на объект `Address`.

Такую связь называют `has-a` - «имеет».

`has-a` означает, что один объект содержит или использует другой объект.
Покупатель не является адресом: покупатель имеет адрес.

```text
Customer has an Address.
Покупатель имеет адрес.
```

Связь `is-a` означает «является». Она описывает не наличие другого объекта,
а принадлежность к более общему типу. Например, менеджер является сотрудником:

```text
Manager is an Employee.
Менеджер является сотрудником.
```

В Java связь `is-a` обычно выражается наследованием, которое будет подробно
изучаться в отдельном модуле. Здесь важно различать две идеи:

- `Customer has-a Address` - покупатель содержит адрес, это композиция;
- `Manager is-a Employee` - менеджер является разновидностью сотрудника, это не композиция.

## Отдельный объект вместо набора полей

Без композиции:

```java
public class Customer {
    private String name;

    // Плохо: устройство адреса смешано с данными Customer.
    private String city;
    private String street;
    private String building;
    private String postalCode;
}
```

Проблемы:

- `Customer` отвечает и за покупателя, и за устройство адреса;
- правила адреса придется хранить внутри покупателя;
- адрес трудно передать отдельно;
- те же поля придется повторить в `Order`, `Warehouse`, `Delivery`.

С отдельным классом:

```java
public class Address {
    // Данные адреса выделены в самостоятельный объект.
    private final String city;
    private final String street;
    private final String building;
    private final String postalCode;

    public Address(
            String city,
            String street,
            String building,
            String postalCode
    ) {
        if (city == null || city.isBlank()) {
            throw new IllegalArgumentException(
                    "City must not be blank"
            );
        }
        if (street == null || street.isBlank()) {
            throw new IllegalArgumentException(
                    "Street must not be blank"
            );
        }
        if (building == null || building.isBlank()) {
            throw new IllegalArgumentException(
                    "Building must not be blank"
            );
        }
        if (postalCode == null || postalCode.isBlank()) {
            throw new IllegalArgumentException(
                    "Postal code must not be blank"
            );
        }

        this.city = city;
        this.street = street;
        this.building = building;
        this.postalCode = postalCode;
    }

    public String getCity() {
        return city;
    }

    public String getStreet() {
        return street;
    }

    public String getBuilding() {
        return building;
    }

    public String getPostalCode() {
        return postalCode;
    }
}
```

Теперь `Address` сам защищает свои правила.

## Валидация составного объекта

Если `Customer` не может существовать без адреса, конструктор должен это проверить:

```java
public class Customer {
    private final String name;
    private final Address address;

    public Customer(String name, Address address) {
        // Customer проверяет собственное имя.
        if (name == null || name.isBlank()) {
            throw new IllegalArgumentException(
                    "Name must not be blank"
            );
        }

        // Обязательная составная часть не может быть null.
        if (address == null) {
            throw new IllegalArgumentException(
                    "Address must not be null"
            );
        }

        this.name = name;
        this.address = address;
    }

    public String getName() {
        return name;
    }

    public Address getAddress() {
        return address;
    }
}
```

`Address` отвечает за корректность адреса. `Customer` отвечает за то, что ссылка на адрес присутствует.

Каждый класс проверяет собственную часть инварианта.

## Вложенные вызовы

Через составной объект можно получить связанные данные:

```java
// Сначала получаем Address, затем читаем его город.
String city = customer.getAddress().getCity();
```

Разбор:

1. `customer.getAddress()` возвращает `Address`.
2. У полученного адреса вызывается `getCity()`.
3. Результатом является строка с городом.

Если конструктор `Customer` запрещает `null` в поле `address`, такой вызов не требует отдельной проверки адреса на `null`.

Это одно из преимуществ сильного инварианта: остальной код делает меньше защитных проверок.

## Зависимость

Зависимость - объект или функция, которые нужны другому объекту для выполнения работы.

### Аналогия из жизни

Зависимость похожа на инструмент, необходимый мастеру. Электрик выполняет
работу сам, но без подходящего измерительного прибора не сможет проверить
проводку. `OrderCalculator` выполняет расчет сам, но использует
`DiscountCalculator` для отдельной части вычисления.

Рассмотрим сервис:

```java
public class OrderCalculator {
    // OrderCalculator зависит от другого объекта.
    private final DiscountCalculator discountCalculator;

    // Зависимость требуется при создании.
    public OrderCalculator(
            DiscountCalculator discountCalculator
    ) {
        this.discountCalculator = discountCalculator;
    }
}
```

`OrderCalculator` зависит от `DiscountCalculator`, потому что использует его при расчете:

```java
public long calculateTotal(
        long subtotalInCents,
        boolean regularCustomer
) {
    long discount = discountCalculator.calculate(
            subtotalInCents,
            regularCustomer
    );

    return subtotalInCents - discount;
}
```

## Создание зависимости внутри класса

Первый вариант:

```java
public class OrderCalculator {
    // Плохо: класс сам жестко выбирает и создает зависимость.
    private final DiscountCalculator discountCalculator =
            new DiscountCalculator();
}
```

Класс сам создает объект, от которого зависит.

Для очень простого кода это работает, но связь становится жесткой:

- нельзя передать заранее настроенный калькулятор;
- способ создания скрыт внутри класса;
- `OrderCalculator` отвечает и за расчет заказа, и за создание помощника;
- заменить помощника без изменения класса нельзя.

## Передача зависимости через конструктор

Передача зависимости через конструктор означает, что нужный объект создается снаружи и передается как аргумент.

```java
// Зависимость создается снаружи.
DiscountCalculator discountCalculator =
        new DiscountCalculator();

// Готовая зависимость передается в конструктор.
OrderCalculator orderCalculator =
        new OrderCalculator(discountCalculator);
```

Сам класс:

```java
public class OrderCalculator {
    private final DiscountCalculator discountCalculator;

    public OrderCalculator(
            DiscountCalculator discountCalculator
    ) {
        // Обязательная зависимость проверяется сразу.
        if (discountCalculator == null) {
            throw new IllegalArgumentException(
                    "Discount calculator must not be null"
            );
        }

        // final-поле получает зависимость один раз.
        this.discountCalculator = discountCalculator;
    }
}
```

Такой прием называется внедрением зависимости через конструктор, или constructor injection.

Внедрение зависимости - передача объекту готовой зависимости извне вместо самостоятельного создания внутри.

Слово injection означает «внедрение» или «передача внутрь». Никакой специальной библиотеки для этого не требуется: это обычный параметр конструктора.

### Аналогия из жизни

Constructor injection похож на выдачу мастеру нужного инструмента до начала
смены. Мастер не обязан самостоятельно покупать и выбирать оборудование:
готовый инструмент передает тот, кто организует работу. Объект так же
получает готовую зависимость при создании и сразу готов выполнять свою задачу.

## Зачем передавать зависимость извне

### Явные требования класса

По конструктору видно, что без `DiscountCalculator` объект работать не может:

```java
new OrderCalculator(discountCalculator);
```

### Корректный объект сразу после создания

Поле объявлено `final` и заполняется в конструкторе:

```java
private final DiscountCalculator discountCalculator;
```

После создания зависимость существует и не может быть случайно заменена.

### Разделение ответственностей

Код запуска решает, какие объекты создать и как связать.

`OrderCalculator` отвечает только за расчет.

### Возможность использовать один объект совместно

```java
DiscountCalculator discounts =
        new DiscountCalculator();

OrderCalculator first =
        new OrderCalculator(discounts);

OrderCalculator second =
        new OrderCalculator(discounts);
```

Оба калькулятора используют один переданный объект.

## Сборка объектов

Сборка объектов - создание экземпляров и связывание их зависимостей.

Для небольшого приложения это можно сделать в `Main`:

```java
public class Main {
    public static void main(String[] args) {
        // Точка сборки создает зависимость.
        DiscountCalculator discountCalculator =
                new DiscountCalculator();

        // Затем связывает объекты через конструктор.
        OrderCalculator orderCalculator =
                new OrderCalculator(discountCalculator);

        // После сборки запускается прикладной сценарий.
        long total = orderCalculator.calculateTotal(
                100_000L,
                true
        );

        System.out.println(total);
    }
}
```

`Main` здесь является точкой сборки:

- создает зависимости;
- передает их нужным объектам;
- запускает сценарий.

Точка сборки - место программы, где создаются основные объекты и устанавливаются связи между ними.

### Аналогия из жизни

Точка сборки похожа на диспетчерскую. Диспетчер знает, какие специалисты и
инструменты нужны, связывает их между собой и затем запускает работу.
Исполнители при этом занимаются своими задачами и не ищут всех помощников
самостоятельно.

## Композиция состояния и композиция поведения

Объект может содержать другую сущность:

```java
class Customer {
    private final Address address;
}
```

Это композиция состояния: адрес является частью модели покупателя.

Объект может содержать сервис:

```java
class OrderCalculator {
    private final DiscountCalculator discountCalculator;
}
```

Это композиция поведения: один объект использует другой для выполнения части операции.

В обоих случаях технический механизм одинаков: поле объектного типа.

## Не передавайте зависимость через setter без необходимости

Плохо:

```java
// Плохо: после пустого конструктора объект еще не готов.
OrderCalculator calculator = new OrderCalculator();

// Обязательную зависимость легко забыть передать.
calculator.setDiscountCalculator(discountCalculator);
```

Между двумя строками существует объект без обязательной зависимости.

Внешний код может забыть вызвать setter:

```java
OrderCalculator calculator = new OrderCalculator();
calculator.calculateTotal(100_000L, true);
```

Если зависимость обязательна, лучше потребовать ее в конструкторе:

```java
// Лучше: невозможно создать OrderCalculator без зависимости.
OrderCalculator calculator =
        new OrderCalculator(discountCalculator);
```

Setter подходит для действительно необязательной или изменяемой настройки, но обязательные зависимости должны быть видны при создании.

### Аналогия из жизни

Передавать обязательную зависимость через setter похоже на отправку
сотрудника на объект без необходимого оборудования с обещанием привезти его
позже. Некоторое время сотрудник существует, но работать не может. Если
инструмент обязателен, его нужно выдать до начала работы, как зависимость в
конструкторе.

## Не создавайте все классы внутри всех классов

Композиция не означает, что каждый объект должен создавать десятки других объектов.

Плохо:

```java
public class OrderCalculator {
    // Плохо: класс самостоятельно создает все зависимости.
    private final DiscountCalculator discountCalculator =
            new DiscountCalculator();
    private final DeliveryCalculator deliveryCalculator =
            new DeliveryCalculator();
    private final TaxCalculator taxCalculator =
            new TaxCalculator();
}
```

Конфигурация - набор выбранных объектов и значений, определяющих, как
работает приложение.

`OrderCalculator` жестко определяет всю конфигурацию.

Лучше передать обязательные зависимости:

```java
public OrderCalculator(
        DiscountCalculator discountCalculator,
        DeliveryCalculator deliveryCalculator,
        TaxCalculator taxCalculator
) {
    // Лучше: точка сборки выбирает и передает готовые объекты.
    this.discountCalculator = discountCalculator;
    this.deliveryCalculator = deliveryCalculator;
    this.taxCalculator = taxCalculator;
}
```

Если параметров конструктора становится слишком много, это может означать, что класс получил слишком широкую ответственность. Тогда нужно вернуться к вопросу: не делает ли он слишком много.

## Полный пример

```java
public class DiscountCalculator {
    private static final int REGULAR_CUSTOMER_PERCENT = 10;

    public long calculate(
            long subtotalInCents,
            boolean regularCustomer
    ) {
        if (!regularCustomer) {
            return 0;
        }

        return subtotalInCents
                * REGULAR_CUSTOMER_PERCENT
                / 100;
    }
}
```

```java
public class OrderCalculator {
    private final DiscountCalculator discountCalculator;

    public OrderCalculator(
            DiscountCalculator discountCalculator
    ) {
        if (discountCalculator == null) {
            throw new IllegalArgumentException(
                    "Discount calculator must not be null"
            );
        }

        this.discountCalculator = discountCalculator;
    }

    public long calculateTotal(
            long subtotalInCents,
            boolean regularCustomer
    ) {
        if (subtotalInCents < 0) {
            throw new IllegalArgumentException(
                    "Subtotal must not be negative"
            );
        }

        long discount = discountCalculator.calculate(
                subtotalInCents,
                regularCustomer
        );

        return subtotalInCents - discount;
    }
}
```

Создание и использование:

```java
DiscountCalculator discounts =
        new DiscountCalculator();

OrderCalculator orders =
        new OrderCalculator(discounts);

long total = orders.calculateTotal(100_000L, true);
```

## Частые ошибки

### Ошибка 1. Оставлять связанные данные отдельными полями во всех классах

Если адрес имеет собственную структуру и правила, его лучше представить объектом `Address`, а не повторять четыре строки в каждой сущности.

### Ошибка 2. Создавать обязательную зависимость через setter

Между конструктором и вызовом setter существует неготовый объект, а вызывающий код может забыть передать зависимость.

### Ошибка 3. Создавать все зависимости внутри рабочего класса

Требования класса становятся скрытыми, а способ сборки невозможно изменить без редактирования самого класса.

### Ошибка 4. Игнорировать длинный конструктор с зависимостями

Большое количество зависимостей часто означает, что класс выполняет слишком много разных задач.

# Состояние, поведение и ответственность класса

## TL;DR

- Состояние объекта - текущие значения его полей.
- Поведение объекта - операции, которые он выполняет через методы.
- Корректный объект должен всегда соблюдать правила своей предметной области.
- Ответственность класса определяет, какие данные и действия относятся к нему.
- Правила лучше хранить рядом с данными, которых они касаются.
- Один класс не должен отвечать сразу за хранение данных, вывод, сохранение и уведомления.
- Открытые поля позволяют обойти правила объекта; эту проблему решает инкапсуляция - ограничение прямого доступа к внутреннему состоянию.

## Что уже известно из Base

В Base вы уже создавали классы с полями и методами:

```java
class Product {
    // Поля хранят состояние конкретного товара.
    String name;
    int price;
    int quantity;

    // Метод использует состояние объекта для расчета.
    int totalPrice() {
        return price * quantity;
    }
}
```

Вы знаете, что:

- класс описывает новый тип данных;
- объект создается через `new`;
- поля хранят состояние объекта;
- методы читают или изменяют это состояние;
- разные объекты одного класса имеют независимые значения полей;
- переменная объектного типа хранит ссылку.

В Core главный вопрос меняется. Теперь недостаточно просто собрать несколько полей и методов в одном классе. Нужно спроектировать класс так, чтобы он:

- отвечал за понятную часть программы;
- не допускал бессмысленного состояния;
- содержал правила работы со своими данными;
- не заставлял остальную программу знать детали его устройства.

## Зачем это нужно

В небольшой задаче некорректное значение можно заметить сразу. В программе из нескольких классов состояние одного объекта меняется из разных участков кода. Если правила нигде не закреплены, появляются отрицательные остатки, пустые имена, неверные суммы и повторяющиеся проверки.

Цель объектной модели - не просто разложить код по классам. Она должна сделать правильное использование простым, а неправильное - невозможным или хотя бы явно обнаруживаемым.

## Объектная модель

Предметная область - это часть реального мира или бизнеса, для которой создается программа. Для интернет-магазина в нее могут входить товары, покупатели, заказы, адреса и оплата.

Объектная модель - это представление предметной области программы через
объекты и связи между ними.

Например:

```text
Покупатель оформляет заказ.
Заказ содержит товары.
У товара есть цена и остаток.
Сервис рассчитывает стоимость доставки.
```

В объектной модели этим понятиям могут соответствовать классы:

```text
Customer
Order
Product
DeliveryCalculator
```

Объектная модель не обязана буквально копировать реальный мир. Она должна отражать только те данные и правила, которые нужны программе.

Например, для доставки может быть важен вес товара, но не цвет упаковки. Значит, вес попадет в модель, а цвет упаковки может не попасть.

## Состояние объекта

Состояние объекта - это совокупность текущих значений его полей.

Рассмотрим товар:

```java
class Product {
    // Все три поля вместе образуют текущее состояние товара.
    String name;
    long priceInCents;
    int quantity;
}
```

Если создан такой объект:

```java
Product product = new Product();

// В Base состояние заполнялось прямой записью в поля.
product.name = "Keyboard";
product.priceInCents = 499_900L;
product.quantity = 8;
```

его состояние состоит из трех значений:

```text
name = "Keyboard"
priceInCents = 499900
quantity = 8
```

Состояние может изменяться во время работы программы. После продажи двух клавиатур значение `quantity` станет равно `6`.

## Допустимое и недопустимое состояние

Не любое сочетание значений имеет смысл.

Для товара могут действовать правила:

- название не должно быть пустым;
- цена не должна быть отрицательной;
- остаток не должен быть отрицательным.

Состояние, которое соответствует правилам класса, называется допустимым или корректным.

Примеры корректного состояния:

```text
name = "Keyboard", priceInCents = 499900, quantity = 8
name = "Mouse", priceInCents = 129900, quantity = 0
```

Примеры некорректного состояния:

```text
name = null, priceInCents = 499900, quantity = 8
name = "", priceInCents = -100, quantity = -5
```

Если поля доступны для прямой записи, любой код может нарушить правила:

```java
product.priceInCents = -500;
product.quantity = -20;
```

Класс в таком виде хранит данные, но не защищает их смысл. В следующих лекциях мы закроем поля и разрешим изменять состояние только контролируемыми способами.

## Поведение объекта

Поведение объекта - это операции, которые объект умеет выполнять.

В Java поведение обычно выражается методами:

```java
class Product {
    String name;
    long priceInCents;
    int quantity;

    // Команда: проверяет правило и меняет состояние.
    boolean sell(int amount) {
        if (amount <= 0 || amount > quantity) {
            return false;
        }

        quantity -= amount;
        return true;
    }

    // Запрос: вычисляет результат, но не меняет состояние.
    long stockValue() {
        return priceInCents * quantity;
    }
}
```

Метод `sell` меняет состояние. Метод `stockValue` только вычисляет результат.

Поведение нужно размещать там, где находятся данные и правила этого поведения. Товар сам знает свой остаток, поэтому проверка возможности продажи естественно находится в `Product`.

Плохо:

```java
// Плохо: внешний код сам отвечает за правило продажи.
if (amount > 0 && amount <= product.quantity) {
    product.quantity -= amount;
}
```

Такое правило может повториться в разных местах программы. Один участок проверит остаток правильно, другой забудет проверить отрицательное количество.

Лучше:

```java
// Лучше: правило продажи находится внутри Product.
boolean sold = product.sell(amount);
```

Теперь правило находится в одном месте.

## Ответственность класса

Ответственность класса - это часть работы программы, за которую отвечает класс.

Для `Product` разумная ответственность может звучать так:

> Хранить корректные данные товара и выполнять операции, непосредственно связанные с состоянием товара.

Для `DeliveryCalculator`:

> Рассчитывать стоимость доставки по параметрам заказа.

Для `Main`:

> Запустить сценарий программы и связать вызовы других объектов.

Ответственность должна быть сформулирована коротко. Если для описания класса требуется длинный список несвязанных действий, скорее всего, класс делает слишком много.

## Один класс - одна основная причина изменения

Полезный ориентир: у класса должна быть одна основная причина измениться.

Рассмотрим класс:

```java
class Product {
    String name;
    long priceInCents;
    int quantity;

    // Относится к состоянию товара.
    boolean sell(int amount) {
        // правила продажи
        return true;
    }

    // Лишняя ответственность: форматирование и вывод.
    void printToConsole() {
        // форматирование и вывод
    }

    // Лишняя ответственность: способ хранения данных.
    void saveToFile() {
        // сохранение данных
    }

    // Лишняя ответственность: отправка уведомлений.
    void notifyAboutLowStock() {
        // отправка уведомления
    }
}
```

У него сразу несколько причин измениться:

- изменились правила товара;
- изменился формат консольного вывода;
- изменился способ хранения данных;
- изменился способ отправки уведомлений.

Это признак слишком широкой ответственности.

Даже если способы хранения и отправки уведомлений еще не изучены, уже важно видеть границу: `Product` не должен отвечать за все действия, в которых участвует товар.

## Сущность и сервис

Сущность - это объект предметной области, у которого есть собственное состояние и смысл.

Примеры сущностей:

- `Product`;
- `Customer`;
- `Order`;
- `BankAccount`.

Сервис - это объект, который выполняет операцию, не принадлежащую естественным образом одной сущности.

Например, расчет доставки может зависеть от заказа, адреса и тарифа. Это отдельная ответственность:

```java
class DeliveryCalculator {
    // Сервис получает все данные через параметры
    // и не хранит состояние товара или заказа.
    long calculate(int weightGrams, int distanceKm) {
        return 20_000L
                + (long) weightGrams * 3
                + (long) distanceKm * 150;
    }
}
```

Слово `service` в этом модуле означает обычный класс с прикладной логикой.

## Команда и запрос

Для понимания методов полезно различать два типа операций.

Команда - метод, который просит объект изменить состояние.

```java
product.sell(2);
```

Запрос - метод, который получает информацию и не должен менять состояние.

```java
int quantity = product.currentQuantity();
long value = product.stockValue();
```

Это не синтаксические термины Java, а способ рассуждать о поведении.

Метод может возвращать результат команды:

```java
boolean sold = product.sell(2);
```

Здесь метод изменяет состояние и сообщает, удалось ли выполнить операцию.

## Почему не каждое действие должно быть методом сущности

Предположим, нужно рассчитать скидку клиента для заказа. В расчете участвуют:

- сумма заказа;
- категория клиента;
- промокод;
- текущая акция.

Если поместить расчет в `Product`, товар получит знания о покупателе и промокодах. Если поместить его в `Customer`, покупатель будет знать устройство заказа.

Разумнее выделить отдельную ответственность:

```java
class DiscountCalculator {
    long calculateDiscount(long totalInCents, boolean regularCustomer) {
        if (regularCustomer) {
            return totalInCents / 10;
        }
        return 0;
    }
}
```

Хорошая объектная модель строится не по правилу «все превратить в класс», а по вопросу «кто должен знать эти данные и соблюдать это правило».

## Пример улучшения класса из Base

Начальный вариант:

```java
class BankAccount {
    // Пока поля открыты, методы можно обойти прямой записью.
    String owner;
    long balanceInCents;
}
```

Внешний код управляет счетом самостоятельно:

```java
// Внешний код самостоятельно меняет состояние счета.
account.balanceInCents += amount;
account.balanceInCents -= payment;
```

Проблемы:

- можно записать отрицательный баланс;
- можно внести отрицательную сумму;
- правила разбросаны по программе;
- класс не отвечает за собственное состояние.

Следующий шаг:

```java
class BankAccount {
    String owner;
    long balanceInCents;

    // Изменение выполняется через метод с проверкой.
    boolean deposit(long amountInCents) {
        if (amountInCents <= 0) {
            return false;
        }

        balanceInCents += amountInCents;
        return true;
    }

    // Снятие не позволяет получить отрицательный баланс.
    boolean withdraw(long amountInCents) {
        if (amountInCents <= 0 || amountInCents > balanceInCents) {
            return false;
        }

        balanceInCents -= amountInCents;
        return true;
    }
}
```

Класс уже лучше управляет состоянием, но прямой доступ к полям все еще позволяет обойти методы:

```java
// Проблема: открытое поле позволяет обойти безопасные методы.
account.balanceInCents = -1_000_000L;
```

Чтобы правила нельзя было обойти, нужна инкапсуляция - ограничение прямого
доступа внешнего кода к внутреннему состоянию объекта.

## Частые ошибки

### Ошибка 1. Считать класс просто набором полей

Если класс только хранит открытые данные, правила работы с ними остаются во внешнем коде и начинают повторяться.

### Ошибка 2. Помещать всю логику в одну сущность

`Product` не должен одновременно считать доставку, печатать отчеты, сохранять данные и отправлять уведомления.

### Ошибка 3. Создавать отдельный класс для каждой маленькой функции

Класс нужен, когда у данных или поведения есть самостоятельный смысл и ответственность. Простая формула не всегда требует нового объекта.

### Ошибка 4. Проверять правило в вызывающем коде

Если правило относится к состоянию товара, счета или заказа, оно должно находиться рядом с этим состоянием.

## Проверь себя

- Что входит в состояние объекта?
- Чем поведение отличается от состояния?
- Что означает корректное состояние?
- Как сформулировать ответственность класса одним предложением?
- Чем сущность отличается от сервиса в рамках этой лекции?
- Почему правило продажи товара лучше хранить в `Product`, а не повторять в `Main`?
- Почему открытые поля позволяют обойти методы класса?

## Что дальше

Мы определили, что объект должен самостоятельно защищать свои правила. В следующей лекции разберем инкапсуляцию, модификаторы `public`, `private`, package-private и границу между внутренним состоянием и публичным контрактом.

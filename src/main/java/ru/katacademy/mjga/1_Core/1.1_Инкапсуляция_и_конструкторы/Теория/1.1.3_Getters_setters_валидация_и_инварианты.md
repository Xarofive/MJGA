# Getters, setters, валидация и инварианты

## TL;DR

- Getter возвращает значение поля и обычно не меняет объект.
- Setter устанавливает значение, но не нужен автоматически для каждого поля.
- Валидация проверяет входные данные до изменения состояния.
- Инвариант - правило, которое должно соблюдаться у корректного объекта постоянно.
- Предметные методы `pay`, `sell`, `deposit` обычно лучше технических setters.
- Неуспешная операция не должна оставлять объект частично измененным.
- Изменяемый объект позволяет менять свое состояние после создания; возвращать такой внутренний объект напрямую опасно для инкапсуляции.

## Что продолжаем из предыдущей лекции

Мы закрыли внутреннее состояние с помощью `private` и определили публичный контракт класса. Теперь нужно решить, какие значения разрешено читать, какие можно менять и где должны находиться проверки.

## Зачем это нужно

Если просто создать getter и setter для каждого поля, класс снова превращается в открытый контейнер данных, только с более длинным синтаксисом. Настоящая защита появляется тогда, когда каждая операция сохраняет правила объекта.

Валидация и инварианты позволяют остальному коду доверять объекту: если объект существует, его состояние корректно.

## Зачем нужны методы доступа

После закрытия полей внешний код больше не может читать и изменять их напрямую:

```java
public class Product {
    // Поля закрыты: прямое чтение и изменение запрещены.
    private String name;
    private long priceInCents;
    private int quantity;
}
```

Это правильно, но объект должен предоставлять необходимые операции.

Для чтения поля часто используется getter. Для изменения поля иногда используется setter.

Эти названия являются соглашением, а не отдельными конструкциями языка Java.

Соглашение - общепринятый способ писать код. Компилятор не требует называть методы `getName` или `setName`, но такие имена понятны Java-разработчикам и инструментам.

## Getter

Getter - метод, который возвращает значение поля.

```java
// Getter дает контролируемый доступ только на чтение.
public String getName() {
    return name;
}
```

Обычно getter:

- не принимает аргументы;
- возвращает значение;
- не меняет состояние объекта.

Для логического поля типа `boolean` часто используется префикс `is`:

```java
private boolean active;

// Для boolean по соглашению используется префикс is.
public boolean isActive() {
    return active;
}
```

Getter дает доступ на чтение, но не дает прямого доступа на запись:

```java
String productName = product.getName();
```

## Setter

Setter - метод, который устанавливает новое значение поля.

```java
// Setter принимает новое значение и записывает его в поле.
public void setName(String newName) {
    name = newName;
}
```

Обычно setter:

- принимает новое значение;
- записывает его в поле;
- возвращает `void`.

Однако setter нужен не для каждого поля. Автоматическое создание setter для всех полей часто разрушает смысл инкапсуляции.

## Почему getter и setter не должны создаваться автоматически

Рассмотрим заказ:

```java
class Order {
    private long totalInCents;
    private boolean paid;

    // Плохо: технический setter не выражает правила заказа.
    public void setTotalInCents(long newTotalInCents) {
        totalInCents = newTotalInCents;
    }

    // Плохо: оплату можно произвольно включать и выключать.
    public void setPaid(boolean newPaid) {
        paid = newPaid;
    }
}
```

Внешний код может выполнить:

```java
// Внешний код может установить бессмысленное состояние.
order.setTotalInCents(-50_000);
order.setPaid(false);
```

Непонятно:

- почему сумма стала отрицательной;
- можно ли отменить оплату простой записью `false`;
- должно ли изменение суммы после оплаты быть разрешено.

Осмысленные операции лучше выражают правила:

```java
class Order {
    private long totalInCents;
    private boolean paid;

    // Предметный метод проверяет, разрешено ли менять сумму.
    public boolean addAmount(long amountInCents) {
        if (paid || amountInCents <= 0) {
            return false;
        }

        totalInCents += amountInCents;
        return true;
    }

    // Метод описывает переход заказа в оплаченное состояние.
    public boolean pay() {
        if (paid || totalInCents <= 0) {
            return false;
        }

        paid = true;
        return true;
    }

    public long getTotalInCents() {
        return totalInCents;
    }

    public boolean isPaid() {
        return paid;
    }
}
```

Методы `addAmount` и `pay` передают намерение вызывающего кода. Setter передает только техническое действие «записать значение».

## Валидация

Валидация - проверка данных на соответствие требованиям.

Примеры требований:

- имя не равно `null`;
- имя не пустое;
- цена неотрицательна;
- количество больше нуля;
- процент находится от `0` до `100`.

Пример:

```java
public boolean changePrice(long newPriceInCents) {
    // Сначала проверяем аргумент.
    if (newPriceInCents < 0) {
        return false;
    }

    // Поле меняется только после успешной проверки.
    priceInCents = newPriceInCents;
    return true;
}
```

Если значение не проходит валидацию, состояние объекта не меняется.

## Проверка строк

Строка может быть непригодной по нескольким причинам:

```java
null
""
"   "
```

`null` означает отсутствие ссылки на объект `String`.

Пустая строка `""` существует, но не содержит символов.

Строка `"   "` содержит пробелы, но часто не считается содержательным названием.

Метод `isBlank()` возвращает `true`, если строка пустая или состоит только из пробельных символов:

```java
private boolean isValidName(String name) {
    // Благодаря && метод isBlank не вызывается для null.
    return name != null && !name.isBlank();
}
```

В выражении используется короткое замыкание `&&`, известное из Base. Если `name == null`, правая часть не выполняется, поэтому `name.isBlank()` не вызывает `NullPointerException`.

## Инвариант объекта

Инвариант - правило, которое должно оставаться истинным для каждого корректно созданного объекта во все доступные внешнему коду моменты.

Для банковского счета инварианты могут быть такими:

```text
owner не равен null и не является пустой строкой
balanceInCents не меньше нуля
```

Для товара:

```text
name заполнено
priceInCents не меньше нуля
quantity не меньше нуля
```

Инвариант отличается от разовой проверки тем, что должен сохраняться постоянно.

Если объект был корректным, публичный метод не должен оставлять его некорректным.

## Как нарушается инвариант

```java
class Product {
    private String name;
    private int quantity;

    // Плохо: проверки отсутствуют.
    public void setName(String newName) {
        name = newName;
    }

    // Плохо: можно записать отрицательный остаток.
    public void setQuantity(int newQuantity) {
        quantity = newQuantity;
    }
}
```

Такой код компилируется:

```java
// Оба вызова нарушают инварианты Product.
product.setName(null);
product.setQuantity(-10);
```

Поля закрыты, но инварианты не защищены.

## Как сохранять инвариант

Каждая точка изменения должна соблюдать правила:

```java
class Product {
    private String name;
    private int quantity;

    // Новое имя проверяется до изменения поля.
    public boolean rename(String newName) {
        if (newName == null || newName.isBlank()) {
            return false;
        }

        name = newName;
        return true;
    }

    // Остаток увеличивается только на положительное значение.
    public boolean addStock(int amount) {
        if (amount <= 0) {
            return false;
        }

        quantity += amount;
        return true;
    }

    // Продажа не позволяет остатку стать отрицательным.
    public boolean sell(int amount) {
        if (amount <= 0 || amount > quantity) {
            return false;
        }

        quantity -= amount;
        return true;
    }
}
```

После любого успешного вызова `quantity` остается неотрицательным.

## Два способа сообщить об ошибке

В простых операциях можно вернуть признак успеха:

```java
public boolean rename(String newName) {
    if (newName == null || newName.isBlank()) {
        return false;
    }

    name = newName;
    return true;
}
```

Другой способ - выбросить исключение.

`IllegalArgumentException` - стандартное непроверяемое исключение Java. Оно
сообщает, что метод получил недопустимый аргумент.

Непроверяемое исключение - исключение, которое компилятор не требует
обязательно обрабатывать через `try/catch` или объявлять через `throws`.

Пример:

```java
public void rename(String newName) {
    if (newName == null || newName.isBlank()) {
        throw new IllegalArgumentException("Name must not be blank");
    }

    name = newName;
}
```

Глубокая модель исключений будет изучаться позже. В этом модуле важно только правило:

- если неуспех является ожидаемым исходом операции, часто удобен `boolean`;
- если вызывающий код нарушил требование метода, подходит `IllegalArgumentException`.

Например, нехватка товара при продаже может быть обычным исходом. Отрицательная цена при создании товара - ошибка входных данных.

## Проверка до изменения

Сначала нужно проверить все условия, затем менять состояние.

Плохо:

```java
public boolean transferTo(BankAccount target, long amountInCents) {
    // Плохо: состояние меняется до завершения всех проверок.
    balanceInCents -= amountInCents;

    if (target == null || amountInCents <= 0 || balanceInCents < 0) {
        return false;
    }

    target.balanceInCents += amountInCents;
    return true;
}
```

Если проверка не пройдет, исходный баланс уже изменен.

Лучше:

```java
public boolean transferTo(BankAccount target, long amountInCents) {
    // Лучше: сначала проверяем все условия операции.
    if (target == null
            || amountInCents <= 0
            || amountInCents > balanceInCents) {
        return false;
    }

    // Изменяем оба объекта только после успешной проверки.
    balanceInCents -= amountInCents;
    target.balanceInCents += amountInCents;
    return true;
}
```

Доступ к `private` полю другого объекта того же класса разрешен. Код класса `BankAccount` может обращаться к приватным полям любого объекта `BankAccount`, а не только текущего объекта.

## Не возвращайте изменяемое внутреннее состояние без необходимости

Пусть класс хранит массив:

```java
class ScoreBoard {
    private int[] scores;

    // Плохо: наружу возвращается ссылка на внутренний массив.
    public int[] getScores() {
        return scores;
    }
}
```

Массив является изменяемым объектом. Изменяемый объект - объект, состояние которого можно изменить после создания.

Внешний код получит ссылку на внутренний массив:

```java
int[] scores = board.getScores();

// Изменение массива снаружи меняет состояние ScoreBoard.
scores[0] = -100;
```

Хотя поле `scores` закрыто, внутреннее состояние изменено через полученную ссылку.

Один из способов защиты - вернуть копию:

```java
import java.util.Arrays;

public int[] getScores() {
    // Лучше: внешний код получает независимую копию.
    return Arrays.copyOf(scores, scores.length);
}
```

`Arrays.copyOf` уже встречался в Base. Здесь он используется как часть инкапсуляции.

Для примитивов и неизменяемых объектов такая проблема обычно не возникает. Например, `String` неизменяем, поэтому безопасно вернуть ссылку на строку:

```java
public String getName() {
    return name;
}
```

## Когда setter уместен

Setter уместен, если:

- поле действительно разрешено изменять напрямую по смыслу;
- новое значение можно полностью проверить внутри метода;
- изменение не требует отдельного предметного действия;
- имя `set...` не скрывает важное бизнес-правило.

Пример:

```java
public void setDeliveryComment(String newDeliveryComment) {
    if (newDeliveryComment == null) {
        deliveryComment = "";
        return;
    }

    deliveryComment = newDeliveryComment;
}
```

Но для операций `pay`, `cancel`, `deposit`, `withdraw`, `sell` лучше использовать предметные названия, а не setters.

## Частые ошибки

### Ошибка 1. Генерировать getter и setter для каждого поля

Каждый метод доступа должен быть обоснован требованиями модели, а не привычкой или возможностью IDE.

### Ошибка 2. Изменять поле до завершения проверки

Если операция завершится отказом после частичного изменения, объект останется в неожиданном состоянии.

### Ошибка 3. Проверять только `null`, но разрешать пустую строку

Для названий и имен часто нужно отклонять также `""` и строки из одних пробелов.

### Ошибка 4. Возвращать внутренний массив напрямую

Внешний код получит ссылку и сможет изменить содержимое без методов класса.

## Что дальше

Публичные методы уже умеют сохранять корректность существующего объекта. Теперь нужно гарантировать, что объект будет корректным с первого момента жизни. Для этого в следующей лекции вводятся конструкторы и ключевое слово `this`.

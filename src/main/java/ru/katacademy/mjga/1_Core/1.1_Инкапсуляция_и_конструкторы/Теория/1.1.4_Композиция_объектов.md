# Композиция объектов

## Результат

После этой лекции вы сможете:

- хранить один объект в поле другого;
- заменить связанный набор полей отдельным классом;
- передать обязательный объект через конструктор;
- собрать простую объектную модель из нескольких классов.

## Когда одного класса недостаточно

Представим заказ с адресом доставки:

```java
public class Order {
    // Плохо: данные адреса смешаны с данными самого заказа.
    private String customerName;
    private String city;
    private String street;
    private String house;
}
```

Поля `city`, `street` и `house` описывают одно понятие - адрес. Если адрес
понадобится покупателю или складу, тот же набор полей придется повторить.

Связанные данные можно выделить в отдельный класс:

```java
public class Address {
    // Поля вместе описывают одно самостоятельное понятие - адрес.
    private final String city;
    private final String street;
    private final String house;

    public Address(String city, String street, String house) {
        // Некорректный адрес не должен быть создан.
        if (city == null || city.isBlank()) {
            throw new IllegalArgumentException("city must not be blank");
        }
        if (street == null || street.isBlank()) {
            throw new IllegalArgumentException("street must not be blank");
        }
        if (house == null || house.isBlank()) {
            throw new IllegalArgumentException("house must not be blank");
        }

        this.city = city;
        this.street = street;
        this.house = house;
    }

    public String getCity() {
        // Getter предоставляет адресные данные только для чтения.
        return city;
    }

    public String getStreet() {
        // Setter отсутствует: улицу нельзя заменить после создания.
        return street;
    }

    public String getHouse() {
        // Возвращаем сохраненный номер дома.
        return house;
    }
}
```

Теперь `Order` хранит объект `Address`:

```java
public class Order {
    // Заказ хранит ссылку на готовый объект Address.
    private final String customerName;
    private final Address deliveryAddress;

    public Order(String customerName, Address deliveryAddress) {
        // Конструктор требует все обязательные части заказа.
        if (customerName == null || customerName.isBlank()) {
            throw new IllegalArgumentException("customer name must not be blank");
        }
        if (deliveryAddress == null) {
            throw new IllegalArgumentException("delivery address is required");
        }

        this.customerName = customerName;
        this.deliveryAddress = deliveryAddress;
    }
}
```

**Композиция** - построение объекта из других объектов. Здесь заказ содержит
адрес доставки. `Address` является **связанным объектом**: он хранится в поле
`Order` и представляет часть модели заказа.

## Обязательные связи через конструктор

Этот раздел учит определять, какие связанные объекты обязательны, и передавать
их в конструктор. Главный результат: после выполнения `new` объект уже
полностью собран и готов к использованию.

Адрес обязателен для заказа, поэтому сначала создается корректный `Address`, а
затем готовый адрес передается в конструктор `Order`:

```java
// Сначала создаем самостоятельный корректный адрес.
Address address = new Address(
        "Moscow",
        "Tverskaya",
        "10"
);

// Затем передаем готовый адрес в обязательный параметр заказа.
Order order = new Order("Alice", address);
```

Конструктор `Order` явно показывает: без имени покупателя и адреса корректный
заказ создать нельзя. Между `Order` и `Address` существует **обязательная
связь**, поэтому адрес передается при создании заказа.

Сравним с другим подходом: сначала создать заказ без адреса, а затем отдельно
установить адрес через setter.

```java
// Плохо: сразу после первой строки заказ еще не имеет адреса.
Order order = new Order("Alice");
order.setDeliveryAddress(address);
```

После первой строки объект `Order` уже существует, но его обязательное поле
`deliveryAddress` равно `null`. Такой заказ создан, хотя еще не готов к
использованию.

Проблема проявится, если код попытается использовать заказ до вызова setter:

```java
Order order = new Order("Alice");

// Ошибка во время выполнения: deliveryAddress пока равен null.
System.out.println(order.getDeliveryCity());
```

Вызов `getDeliveryCity()` обращается к `deliveryAddress.getCity()`. Поскольку
адрес еще не установлен, программа завершится с `NullPointerException`.

Даже если разработчик планирует вызвать setter следующей строкой, другой код
может получить объект раньше, произойдет исключение или вызов setter просто
забудут написать. Класс `Order` сам не гарантирует, что созданный заказ
пригоден для работы.

Передача адреса через конструктор устраняет промежуточное некорректное
состояние:

```java
// Order нельзя создать, не передав обязательный адрес.
Order order = new Order("Alice", address);

// Адрес уже установлен, поэтому вызов безопасен.
System.out.println(order.getDeliveryCity());
```

Setter уместен, когда значение действительно разрешено менять после создания
объекта. Например, пользователь может позднее выбрать необязательный
комментарий к доставке:

```java
Order order = new Order("Alice", address);

// Комментарий необязателен: заказ корректен и без него.
order.setDeliveryComment("Позвонить за 10 минут");
```

Разница следующая:

- без адреса заказ нельзя доставить, поэтому адрес передается в конструктор;
- без комментария заказ остается корректным, поэтому комментарий можно
  добавить позже отдельным методом.

## Вложенные вызовы

`Order` может предоставить нужную информацию, не открывая собственные поля:

**Вложенный вызов** - вызов метода одного объекта через ссылку, которая
хранится в другом объекте.

```java
public String getDeliveryCity() {
    // Order делегирует чтение города объекту Address.
    return deliveryAddress.getCity();
}
```

Или вернуть сам адрес для чтения:

```java
public Address getDeliveryAddress() {
    // Возвращаем неизменяемый в этой модели объект Address.
    return deliveryAddress;
}
```

В нашем примере поля `Address` объявлены `final`, а setters отсутствуют,
поэтому после создания адрес не меняется.

## Композиция поведения

Объект может не только хранить другой объект, но и использовать его работу.
Добавим товар в строку заказа:

```java
public class OrderItem {
    // Строка заказа содержит ссылку на товар, а не копию его полей.
    private final Product product;
    private final int amount;

    public OrderItem(Product product, int amount) {
        // Обязательный товар и количество проверяются при создании.
        if (product == null) {
            throw new IllegalArgumentException("product is required");
        }
        if (amount <= 0) {
            throw new IllegalArgumentException("amount must be positive");
        }

        this.product = product;
        this.amount = amount;
    }

    public long totalPrice() {
        // Используем публичный контракт Product для расчета суммы.
        return product.getPriceInCents() * amount;
    }
}
```

Для этого в `Product` нужен getter цены:

```java
public long getPriceInCents() {
    // Цена доступна для расчета, но не для прямого изменения.
    return priceInCents;
}
```

`OrderItem` не копирует название и цену товара в отдельные поля. Он хранит
ссылку на `Product` и использует его **публичный контракт** - методы,
доступные внешнему коду.

## Как выделять отдельный класс

Отдельный класс обычно полезен, если данные:

- описывают самостоятельное понятие;
- всегда используются вместе;
- имеют собственные правила проверки;
- повторяются в разных местах программы.

Не нужно превращать каждое поле в отдельный класс. Цель композиции - сделать
модель понятнее, а не увеличить количество файлов.

## Итог модуля

Нормальный объектный класс:

1. Закрывает внутренние поля.
2. Создается в корректном состоянии.
3. Проверяет изменения до записи.
4. Не предоставляет setters без необходимости.
5. Использует `static` только для общих данных.
6. Собирается из других объектов, когда данные образуют отдельное понятие.

Этого достаточно, чтобы перейти к практике: переписать простые классы Base в
инкапсулированную модель и собрать небольшой заказ из товара и адреса.

## Словарь

- **Композиция** - построение объекта из других объектов.
- **Связанный объект** - объект, который хранится в поле другого объекта и
  представляет часть его модели.
- **Обязательная связь** - связь, без которой объект не считается корректным;
  обычно передается через конструктор.
- **Вложенный вызов** - вызов метода одного объекта через ссылку, хранящуюся
  в другом объекте.
- **Публичный контракт** - доступные внешнему коду методы класса.

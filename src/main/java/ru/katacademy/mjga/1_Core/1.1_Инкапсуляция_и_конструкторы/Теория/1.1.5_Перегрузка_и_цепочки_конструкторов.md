# Перегрузка и цепочки конструкторов

## TL;DR

- Перегрузка позволяет объявить несколько конструкторов с разными параметрами.
- Перегрузки различаются количеством, типами или порядком параметров.
- Сигнатура описывает имя и типы параметров; имена самих параметров в нее не входят.
- `this(...)` вызывает другой конструктор того же класса.
- Вызов `this(...)` должен быть первой инструкцией конструктора.
- Общую валидацию лучше сосредоточить в одном полном конструкторе.
- Короткий конструктор допустим только при понятных значениях по умолчанию.

## Что продолжаем из предыдущей лекции

Один конструктор позволяет создать объект сразу в корректном состоянии. Иногда часть начальных данных действительно необязательна, и класс должен поддерживать несколько понятных способов создания.

## Зачем это нужно

Без перегрузки внешний код вынужден передавать технические значения-заглушки или создавать объект через setters. Но если просто скопировать код нескольких конструкторов, проверки начнут дублироваться.

Цепочки конструкторов дают удобные варианты вызова и одновременно сохраняют одно место, где выполняются полная валидация и присваивание полей.

## Зачем классу несколько конструкторов

Иногда объект можно корректно создать несколькими способами.

Например, товар может быть создан:

- с названием и ценой, но без начального остатка;
- с названием, ценой и известным остатком.

Оба состояния допустимы:

```java
// Создание без начального остатка.
new Product("Keyboard", 499_900L);

// Создание с явно указанным остатком.
new Product("Keyboard", 499_900L, 8);
```

Для этого класс может объявить несколько конструкторов.

## Перегрузка

Перегрузка - объявление нескольких методов или конструкторов с одним именем, но разными списками параметров.

Для конструкторов имя всегда совпадает с именем класса:

```java
// Перегрузка с двумя параметрами.
public Product(String name, long priceInCents) {
    // ...
}

// Перегрузка с тремя параметрами.
public Product(String name, long priceInCents, int quantity) {
    // ...
}
```

Список параметров отличается количеством параметров.

Перегрузка возможна и при одинаковом количестве параметров, если отличаются их типы:

```java
public Product(String name, long priceInCents) {
    // ...
}

public Product(String name, String priceText) {
    // ...
}
```

Но такие варианты должны иметь понятный смысл. Не следует добавлять перегрузки только потому, что язык это позволяет.

### Аналогия из жизни

Перегрузка похожа на несколько форм оформления одного заказа. В короткой
форме указываются только обязательные данные, а в полной можно дополнительно
указать адрес или комментарий. Назначение остается одним, но набор входных
данных различается.

## Сигнатура

Сигнатура метода - имя метода и типы его параметров в определенном порядке.

Для конструкторов при выборе перегрузки важны:

- количество параметров;
- типы параметров;
- порядок параметров.

Имена параметров не различают перегрузки.

### Аналогия из жизни

Сигнатуру можно сравнить с описанием нужного окна обслуживания: важны тип и
порядок документов, которые туда подают. Как посетитель назвал документы у
себя в папке, роли не играет. Компилятор также различает перегрузки по типам
и порядку параметров, а не по их именам.

Нельзя:

```java
// Эти конструкторы нельзя объявить вместе:
// имена параметров не входят в сигнатуру.
public Product(String name, long price) {
}

public Product(String title, long cost) {
}
```

Для компилятора обе сигнатуры одинаковы:

```text
Product(String, long)
```

## Как Java выбирает конструктор

Java смотрит на переданные аргументы:

```java
new Product("Keyboard", 499_900L);
```

Подходит:

```java
Product(String, long)
```

Для вызова:

```java
new Product("Keyboard", 499_900L, 8);
```

подходит:

```java
Product(String, long, int)
```

Выбор подходящей перегрузки выполняется компилятором.

## Проблема дублирования

Плохой вариант:

```java
public class Product {
    private String name;
    private long priceInCents;
    private int quantity;

    public Product(String name, long priceInCents) {
        if (name == null || name.isBlank()) {
            throw new IllegalArgumentException(
                    "Name must not be blank"
            );
        }
        if (priceInCents < 0) {
            throw new IllegalArgumentException(
                    "Price must not be negative"
            );
        }

        this.name = name;
        this.priceInCents = priceInCents;
        this.quantity = 0;
    }

    public Product(String name, long priceInCents, int quantity) {
        if (name == null || name.isBlank()) {
            throw new IllegalArgumentException(
                    "Name must not be blank"
            );
        }
        if (priceInCents < 0) {
            throw new IllegalArgumentException(
                    "Price must not be negative"
            );
        }
        if (quantity < 0) {
            throw new IllegalArgumentException(
                    "Quantity must not be negative"
            );
        }

        this.name = name;
        this.priceInCents = priceInCents;
        this.quantity = quantity;
    }
}
```

Проверки имени и цены повторяются. Если правило изменится, придется исправлять два места.

## Вызов одного конструктора из другого

Один конструктор того же класса можно вызвать через `this(...)`.

```java
public Product(String name, long priceInCents) {
    // Вместо копирования проверок вызываем полный конструктор.
    this(name, priceInCents, 0);
}

public Product(String name, long priceInCents, int quantity) {
    // Вся общая валидация остается в одном месте.
    if (name == null || name.isBlank()) {
        throw new IllegalArgumentException(
                "Name must not be blank"
        );
    }
    if (priceInCents < 0) {
        throw new IllegalArgumentException(
                "Price must not be negative"
        );
    }
    if (quantity < 0) {
        throw new IllegalArgumentException(
                "Quantity must not be negative"
        );
    }

    this.name = name;
    this.priceInCents = priceInCents;
    this.quantity = quantity;
}
```

Запись:

```java
this(name, priceInCents, 0);
```

вызывает другой конструктор текущего класса.

Не путайте:

```java
// this.name обращается к полю текущего объекта.
this.name
```

и:

```java
// this(...) вызывает другой конструктор текущего класса.
this(...)
```

`this.name` - обращение к полю текущего объекта.

`this(...)` - вызов другого конструктора текущего класса.

## Цепочка конструкторов

Цепочка конструкторов - последовательность вызовов, в которой один конструктор передает работу другому.

Пример:

```java
public Product(String name) {
    // Первый шаг цепочки: подставляем цену по умолчанию.
    this(name, 0);
}

public Product(String name, long priceInCents) {
    // Второй шаг: подставляем нулевой остаток.
    this(name, priceInCents, 0);
}

public Product(String name, long priceInCents, int quantity) {
    // Последний шаг: общая валидация и присваивание.
}
```

Вызов:

```java
new Product("Keyboard");
```

образует цепочку:

```text
Product(String)
-> Product(String, long)
-> Product(String, long, int)
```

Последний конструктор выполняет общую проверку и инициализацию.

### Аналогия из жизни

Цепочка конструкторов похожа на несколько окон, которые передают документы
одному главному специалисту. Первые окна дополняют отсутствующие сведения,
но окончательная проверка выполняется в одном месте. Благодаря этому правила
не копируются между всеми вариантами оформления.

## `this(...)` должен быть первой инструкцией

Вызов другого конструктора должен находиться первой инструкцией конструктора:

```java
public Product(String name, long priceInCents) {
    // Правильно: this(...) является первой инструкцией.
    this(name, priceInCents, 0);
}
```

Нельзя:

```java
public Product(String name, long priceInCents) {
    System.out.println("Creating product");

    // Ошибка: перед this(...) уже выполнена другая инструкция.
    this(name, priceInCents, 0); // ошибка компиляции
}
```

Это правило гарантирует, что инициализация объекта идет по одной понятной цепочке.

## Основной конструктор

Основной конструктор - не официальный термин Java, а удобное название конструктора, в котором сосредоточены полная валидация и присваивание полей.

В примере основным является:

```java
public Product(String name, long priceInCents, int quantity) {
    // полная проверка
    // присваивание всех полей
}
```

Остальные конструкторы подставляют значения по умолчанию и делегируют ему работу.

Делегирование - передача части работы другому методу или конструктору.

## Значение по умолчанию в предметной области

Не каждое техническое значение по умолчанию является разумным для задачи.

Например:

```java
public Customer() {
    this.name = null;
}
```

`null` - стандартное значение поля, но покупатель без имени может быть недопустим.

Для товара отсутствие остатка может естественно означать нулевой остаток:

```java
public Product(String name, long priceInCents) {
    this(name, priceInCents, 0);
}
```

Перед добавлением короткого конструктора нужно спросить:

- действительно ли пропущенное значение необязательно;
- существует ли безопасное и понятное значение по умолчанию;
- не скрывает ли перегрузка ошибку вызывающего кода.

### Аналогия из жизни

Значение по умолчанию похоже на стандартную комплектацию товара. Если клиент
не выбрал цвет упаковки, можно использовать обычную упаковку. Но нельзя
самостоятельно подставить случайный адрес доставки: отсутствие адреса
означает, что обязательных данных не хватает.

## Не создавайте неоднозначные конструкторы

Плохо:

```java
new Product("Keyboard", 100, 20);
```

По вызову непонятно:

- `100` - цена или вес;
- `20` - количество или скидка.

Имена параметров видны в объявлении, но не всегда очевидны в месте вызова.

Можно улучшить типы и порядок:

```java
new Product("Keyboard", 499_900L, 8);
```

Но если у класса появляется слишком много похожих параметров, конструктор становится трудным для чтения. В дальнейшем существуют дополнительные способы создания сложных объектов, но в текущем модуле лучше:

- оставлять только обязательные данные;
- не создавать десятки перегрузок;
- группировать связанные данные в отдельные объекты, если у них есть собственный смысл.

## Перегрузка конструктора и setter решают разные задачи

Конструктор создает объект:

```java
Product product = new Product("Keyboard", 499_900L);
```

Метод изменяет уже существующий объект:

```java
product.changePrice(449_900L);
```

Не следует создавать объект без обязательных данных только ради последующих setters:

```java
Product product = new Product();
product.setName("Keyboard");
product.setPriceInCents(499_900L);
```

Так объект некоторое время находится в неполном состоянии.

## Полный пример

```java
public class Product {
    private String name;
    private long priceInCents;
    private int quantity;

    public Product(String name, long priceInCents) {
        this(name, priceInCents, 0);
    }

    public Product(String name, long priceInCents, int quantity) {
        if (name == null || name.isBlank()) {
            throw new IllegalArgumentException(
                    "Name must not be blank"
            );
        }
        if (priceInCents < 0) {
            throw new IllegalArgumentException(
                    "Price must not be negative"
            );
        }
        if (quantity < 0) {
            throw new IllegalArgumentException(
                    "Quantity must not be negative"
            );
        }

        this.name = name;
        this.priceInCents = priceInCents;
        this.quantity = quantity;
    }

    public String getName() {
        return name;
    }

    public long getPriceInCents() {
        return priceInCents;
    }

    public int getQuantity() {
        return quantity;
    }
}
```

## Частые ошибки

### Ошибка 1. Пытаться различить перегрузки именами параметров

`Product(String name, long price)` и `Product(String title, long cost)` имеют одинаковую сигнатуру.

### Ошибка 2. Копировать проверки во все конструкторы

Дублирование приводит к расхождению правил при следующем изменении.

### Ошибка 3. Писать код перед `this(...)`

Вызов другого конструктора должен быть первой инструкцией.

### Ошибка 4. Создавать слишком много похожих перегрузок

Если вызов состоит из нескольких одинаковых чисел и его смысл не читается, API класса становится опасным.

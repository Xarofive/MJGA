# Git перед проектной работой

## TL;DR

- Git хранит историю изменений проекта.
- Рабочий цикл: изменить файлы -> проверить `git status` -> посмотреть `git diff` -> добавить `git add` -> сохранить `git commit`.
- Коммит должен быть маленьким и осмысленным.
- `.gitignore` исключает из истории временные файлы, сборку и настройки IDE.
- Git подробно нужен ближе к проектной работе, когда важно сохранять историю изменений и объяснять ход разработки.

## Зачем Git перед проектной работой

В обычных задачах платформы достаточно отправить решение на проверку автотестами. В проектной работе ситуация другая: появляется больше файлов, несколько этапов разработки, README, настройки, тесты и постепенные изменения. Без истории изменений трудно понять:

- что именно изменилось;
- когда появилась ошибка;
- какая задача была сделана в конкретный момент;
- можно ли безопасно откатить идею;
- почему код сейчас выглядит именно так.

Git можно представить как журнал разработки. Он не пишет код за вас, но сохраняет важные состояния проекта.

## Репозиторий

Репозиторий - это папка проекта, внутри которой Git хранит историю.

Создать репозиторий:

```bash
git init
```

После этого в проекте появится скрытая папка `.git`. Ее не нужно редактировать руками.

Проверить состояние:

```bash
git status
```

`git status` - команда, которую стоит запускать часто. Она отвечает на вопрос: "Что изменилось и что Git сейчас видит?"

## Три состояния файла

Упрощенно файл может быть в одном из трех состояний:

1. Изменен в рабочей папке.
2. Добавлен в область подготовки через `git add`.
3. Сохранен в истории через `git commit`.

Пример:

```bash
git status
git add src/main/java/ru/katacademy/core/app/Main.java
git status
git commit -m "Add application entry point"
```

`git add` не сохраняет историю. Он только готовит файлы к коммиту.

`git commit` создает точку истории.

## `git diff`

Перед коммитом полезно посмотреть изменения:

```bash
git diff
```

Команда показывает, какие строки добавлены, изменены или удалены.

Если файл уже добавлен через `git add`, обычный `git diff` может ничего не показать. Тогда используйте:

```bash
git diff --staged
```

Смысл простой: перед коммитом нужно понимать, что именно вы сохраняете.

## Первый рабочий цикл

1. Создайте или измените файл.
2. Выполните:

```bash
git status
```

3. Посмотрите изменения:

```bash
git diff
```

4. Добавьте нужные файлы:

```bash
git add src/main/java/ru/katacademy/core/model/Product.java
```

5. Еще раз проверьте:

```bash
git status
git diff --staged
```

6. Сделайте коммит:

```bash
git commit -m "Add product model"
```

## Как писать сообщения коммитов

Хорошее сообщение коротко говорит, что сделано.

Хороший пример:

```text
Add product model
Implement delivery price calculation
Fix integer overflow in total price
Move order logic to service package
```

Плохой пример:

```text
changes
fix
lesson
my work
final
```

Коммит читают как запись в истории. Через неделю сообщение `fix` не объяснит ничего.

## Маленькие коммиты

Маленький коммит содержит одну логическую идею.

Пример хорошей последовательности:

```text
Add product and customer models
Add delivery calculator
Handle invalid delivery input
Add README with run instructions
```

Плохой пример коммита:

```text
Add all module 1 project
```

Почему это плохой пример:

- трудно проверить;
- трудно найти ошибку;
- трудно понять, что изменилось;
- откатить одну часть сложнее.

## `.gitignore`

Не все файлы проекта нужно сохранять в Git. Например:

- скомпилированные `.class`;
- папки сборки `target`, `build`, `out`;
- временные файлы;
- локальные настройки IDE;
- секреты, токены, пароли.

Пример `.gitignore` для учебного Java-проекта:

```gitignore
out/
target/
build/
*.class
.idea/
*.iml
```

Иногда `.idea` хранят в репозитории частично, но на старте проще исключить локальные настройки IDE.

Важно: `.gitignore` не удаляет файл из истории, если файл уже был закоммичен. Он только предотвращает добавление новых подходящих файлов.

## `git log`

Историю можно посмотреть так:

```bash
git log
```

Краткий вариант:

```bash
git log --oneline
```

Пример:

```text
a1b2c3d Add delivery calculator
9f8e7d6 Add product model
123abcd Add application entry point
```

## Что делать перед задачей

Перед началом работы:

```bash
git status
```

Если есть незакоммиченные изменения, нужно понять, ваши они или нет. В командной работе нельзя случайно смешивать чужие изменения со своими.

Для учебного процесса правило такое: одна задача - один или несколько осмысленных коммитов.

## Что не нужно делать новичку без понимания

Не используйте команды, которые удаляют изменения, если не понимаете последствий:

**Важно:** команды ниже могут удалить незакоммиченную работу. Не запускайте их "наугад" и не используйте как быстрый способ убрать ошибки.

```bash
git reset --hard
git clean -fd
```

Эти команды могут уничтожить незакоммиченную работу.

На старте работы с Git достаточно безопасного набора:

```bash
git init
git status
git diff
git add
git commit
git log
```

## Пример истории для учебного проекта

Учебный проект `order-management-core` можно вести так:

```text
1. Add project structure
2. Add order and product models
3. Implement order total calculation
4. Handle invalid order input
5. Add README with examples
```

Каждый коммит соответствует понятному шагу.

## Частые ошибки

### Ошибка 1. Коммит всего подряд

```bash
git add .
git commit -m "fix"
```

`git add .` не запрещен, но перед ним нужно посмотреть `git status` и `git diff`, чтобы случайно не добавить мусор.

### Ошибка 2. Нет `.gitignore`

В историю попадают `out/`, `.class`, временные файлы. Репозиторий засоряется.

### Ошибка 3. Один огромный коммит

Если в одном коммите модели, сервисы, исправления, README и форматирование, такой коммит трудно ревьюить.

### Ошибка 4. Коммит без проверки запуска

Перед коммитом желательно хотя бы запустить программу или тесты, если они есть.

## Практика

### Задание 1. Создать репозиторий

Создайте учебную папку `order-management-core` и выполните:

```bash
git init
git status
```

### Задание 2. Первый коммит

Создайте `README.md` с коротким описанием проекта.

Выполните:

```bash
git status
git diff
git add README.md
git commit -m "Add project README"
```

### Задание 3. `.gitignore`

Создайте `.gitignore`:

```gitignore
out/
target/
build/
*.class
.idea/
*.iml
```

Сделайте отдельный коммит:

```bash
git add .gitignore
git commit -m "Add Java gitignore"
```

### Задание 4. Три осмысленных коммита

Добавьте Java-код по шагам:

1. `Main` с запуском.
2. Класс или метод расчета заказа.
3. Проверку некорректных данных.

На каждый шаг сделайте отдельный коммит.

### Задание 5. Показать историю

Выполните:

```bash
git log --oneline
```

Проверьте, что по сообщениям коммитов понятна история работы.

- Что хранит `.gitignore`?
- Почему маленькие коммиты лучше огромного коммита?
- Какая команда показывает текущее состояние?
- Какая команда показывает историю?
